## 分析問題
將「火水風土雷無」+ 克制關係，改為「戰士/法師」等職業標籤（無克制），是否更符合目前的遊戲設計？

---

## 一、現行系統的實際運作（程式碼層面）

### 傷害計算的兩層結構（game.js L2366-2380）

```
傷害 × elemAdv(tower.dmgType vs enemy.elem)  ← 克制乘數
     × (1 - armor)
     × (1 - enemy.resist[tower.elem])         ← 抗性減免
```

這是**兩個獨立機制**，常被混淆：

| 機制 | 觸發條件 | 數值 | 角色 |
|------|---------|------|------|
| `elemAdv` | 塔的 dmgType × **怪物的 elem 屬性** | ×1.3 / ×0.7 | 克制加成 |
| `resist` | 怪物的抗性字典 × **塔的 elem** | −40% / −50% | 傷害減免 |

## 二、克制系統在現行波次設計中的實際影響

逐章分析 waves.js：

| 章 | 波次 | enemy.elem 分佈 | resist 設計 | elemAdv 是否有效 |
|----|------|----------------|-------------|----------------|

| Ch2 (W5-8) | W5 | fire | 無 | ✅ 水打火 ×1.3 有效 |
| Ch2 | W6 | wind | resist: {water:0.4} | 🔶 water 抗有意義，elemAdv 有效 |

| Ch2 Boss | W8 | fire | 無 | ✅ 有效但差異小 |
| Ch3 (W9-12) | W9,W11 | 無 | random_dual | ❌ 隨機抗性≠策略 |
| Ch3 | W10 | wind | 無 | ✅ 有效但玩家已有 2 個元素，選擇受限 |
| Ch3 Boss | W12 | 無 | random_dual | ❌ |
| Ch4 (W13-16) | W13,W14 | 無 | **全元素均等** (25-30%) | ❌ 均等抗性=無差異 |
| Ch4 | W15 | fire | random_dual | ❌ 隨機 |

### 關鍵發現
**克制關係（elemAdv）在 20 波中只有 4-5 波實質有效（Ch2）。**
Ch3 起全面轉向「隨機抗」或「均等抗」，元素選擇對波次應對的影響幾乎消失。

## 三、元素系統的真實用途（現行設計）

元素目前主要服務的不是克制，而是：

### 1. 升級路徑分歧（最核心用途）
```
Lv2 基礎塔 → 選元素 → Lv3 元素塔
Lv3 + 第2元素 → INFUSIONS → Lv4 雙屬塔（36種）
Lv4 + 第3元素 → TRIPLE → Lv5 三屬塔（20種）
Lv4 同屬×2 → PURE → Lv5/Lv6 純屬塔（6種）
```
元素是**建造路徑的索引鍵**，不是克制標籤。

### 2. 技能身份（Lv4 起才生效）
- 🔥 火：burn / ignite / detonate / ramp → Boss Killer
- 💧 水：chill / freeze / zone_slow → Group Handler
- 🌪️ 風：execute / knockback → Finisher

這些技能才是元素差異的真正來源，而非 ×1.3/×0.7 的克制數值。

### 3. Lv3 的尷尬現狀
Lv3（ELEM_BASE）：
- `skills: []`（沒有任何技能）
- dmgType 有，但 Ch1 後怪物大多無 elem
- 現行數值：各元素 Lv3 完全相同（damage/atkSpd/range 一樣）

**結論：Lv3 的元素差異目前只剩圖示和名字。**

## 四、職業標籤系統（Warrior/Mage 式）評估

### 概念設計
- `ELEM_KEYS` → `CLASS_KEYS`: ['warrior', 'mage', 'ranger', 'support', 'titan', 'void']
- 無 `elemAdv` 克制矩陣
- INFUSIONS：職業組合塔（warrior+mage = 劍聖、mage+ranger = 狙魔師）
- 波次設計：只有 armor 和 HP，無克制關係

### 得到的

| 項目 | 現況 | 職業標籤後 |
|------|------|-----------|
| 設定複雜度 | 高（6×6 克制矩陣，需驗證完整性） | 低（無矩陣） |
| editor-validation | 需要 step1/step2 克制警示 | 不需要 |
| 波次設計自由度 | 受限（需考慮哪些元素有抗性） | 高（只有 armor） |
| PVP 同步複雜度 | 需同步 elemAdv 設定 | 不需要 |
| 玩家策略層次 | 「選元素克制波次」 | 「選職業組合克制對手送兵」|

### 失去的

| 項目 | 影響程度 | 說明 |
|------|---------|------|
| 波次弱點策略 | **低**（現行波次 Ch3 起已無意義） | 克制設計未被波次充分利用 |
| 元素幻想感 | 中（風格選擇） | 火水風 vs 戰士法師 是主題差異 |
| 既有數值資料 | **中高**（ELEM_BASE/INFUSIONS 需全面重命名） | 程式碼改動大 |

## 五、深層設計問題：兩個系統的根本差異

### 「選元素克制波次」→ 被動適應型
- 設計邏輯：「看波次弱點，選對應元素」
- 問題：Ch3 起改用隨機/均等抗性，玩家無從適應
- PVE 中此邏輯失效；在 PVP 中波次相同，克制更無意義

### 「選職業組合克制對手送兵」→ 主動博弈型
- 設計邏輯：「選我的塔型來 counter 對手送兵的類型」
- PVP 核心：知道對手喜歡送重甲騎士 → 選有碎甲/易傷的元素組合
- 與 LTW 機制天然契合

### 結論：職業標籤更符合 PVP-LTW 設計目標

現行元素克制系統是**單人PvE波次的設計語言**，而本作的核心是**LTW送兵對抗**。職業標籤讓策略從「對抗波次」轉為「對抗對手的送兵選擇」，與遊戲定位更一致。

## 六、可行性：需要改動什麼

### 小改動（只有 key 重命名，邏輯不動）

如果只是把 `fire→warrior, water→mage, wind→ranger...`：
- `js/towers.js`：ELEM/ELEM_KEYS/ELEM_BASE/INFUSIONS/TRIPLE/PURE 重命名 → **最大工作量**
- `js/waves.js`：enemy.elem 和 resist 字典的 key → 波次數量少，快速
- `js/config.js`：elemAdv 刪除 → 一行
- `js/game.js`：邏輯完全不動，只改字串 key → 小
- `skill-editor.html`：ELEM display name 改 → 小
- `index.html`：沙盒的 elem 按鈕 → 小

### 中改動（移除 elemAdv 計算）
- `game.js L2367-2370`：刪除 elemAdv 乘數 → 4 行
- `config.js`：刪除 elemAdv 物件 → 5 行
- `skill-editor.html`：刪除克制矩陣 UI → 減少複雜度

### 影響最小的路徑（推薦評估方向）
**不改名，而是把 elemAdv 清空（全部設為 1.0 或不設定）**

→ 效果等同「無克制」的職業標籤系統，但不需要重命名任何 key。
→ 驗證「去掉克制後遊戲是否好玩」後再決定要不要正式重命名。
→ 可透過現有 skill-editor 的 elemAdv 矩陣直接操作，零程式碼改動。

## 七、綜合評估

| 評估維度 | 保留元素克制 | 職業標籤（無克制） |
|---------|------------|-----------------|
| MVP 驗證目標（LTW+升級路徑）| ✅ 均能達成 | ✅ 均能達成 |
| 波次設計複雜度 | 高（需設計克制關係） | 低 |
| 策略深度（PvE）| 中（Ch2 有效，Ch3 後失效） | 相同（靠技能和 armor）|
| 策略深度（PVP）| 低（克制 AI 波次無意義） | **更高**（克制對手送兵）|
| 設定與維護成本 | 高（克制矩陣、驗證、PVP 同步）| 低 |
| 程式碼改動量 | 不變 | 中（主要是重命名）|
| 未來擴展性 | 需維護克制完整性 | 自由擴展職業 |

### 建議

**階段1（立即可做）**：
在 skill-editor 中將 `elemAdv` 全部清空（所有乘數設為 1.0），繼續進行 elem-filter-editor 計畫，驗證「無克制的元素組合系統」是否玩起來合理。此時元素名稱不變，但行為等同職業標籤。

**階段2（驗證後決定）**：
- 若「無克制」的遊戲感覺缺少什麼 → 重新設計克制關係（可能改為職業克制送兵類型，而非波次元素）
- 若「無克制」感覺完整 → 做正式重命名（fire→戰士/弓手/法師，根據技能身份命名）

**不建議**：在 MVP 驗證前就全面重命名，因為重命名工作量大（ELEM_BASE/INFUSIONS 36 個、TRIPLE 20 個），且驗證方向尚未確認。

## 八、若決定重命名，推薦的職業對應

基於現有技能身份（技能是真正的差異來源）：

| 原元素 | 核心技能 | 建議職業名 | 設計定位 |
|--------|---------|-----------|---------|
| 🔥 火 | burn / ignite / detonate / ramp | 劊子手 / 焚獵者 | Boss Killer，集火單體 |
| 💧 水 | chill / freeze / zone_slow | 霜術師 / 控場師 | Group Handler，群體控制 |
| 🌪️ 風 | execute / knockback / pierce | 狙擊手 / 終結者 | Finisher，擊殺低HP |
| ⛰️ 土 | shred / armorStack / aura | 破甲師 / 鐵衛 | Tank Buster，護甲削減 |
| ⚡ 雷 | chain / multishot / unstable | 鏈電師 / 爆破手 | AoE Spread，鏈式傷害 |
| ⬜ 無 | lifedrain / interest / wealthScale | 行商 / 支援 | 經濟/輔助，特殊功能 |

命名方向：**以技能角色命名比以傳統元素命名更清楚**（玩家選「霜術師」就知道選的是控場，不需要記住「水克火」）。
